No hype, no hand-waving. 
Does the system read code snippets? 
Yes — but selectively and purposefully 
The system does not “understand” all code line-by-line like a human. 
Instead, it reads targeted snippets to extract signals. 
What it reads (examples) 
From a GitHub repo, the system fetches: 
 README.md (primary intent & scope) 
 File tree (structure) 
 Selected files based on heuristics, e.g.: 
o migrations/*.sql 
o schema.ts, models.py 
o middleware/* 
o Dockerfile 
o tests/* 
 Commit messages and metadata 
Then it reads snippets, not entire files. 
Example snippet read: 
CREATE TABLE payments ( 
id UUID PRIMARY KEY, 
amount INT NOT NULL, 
status VARCHAR(20), 
created_at TIMESTAMP 
); 
From this snippet, it extracts signals like: 
 schema presence 
 normalization hints 
 migration ownership 
Does it understand project structure? 
Yes — structure is one of the strongest signals 
Project structure is read via the file tree, not deep parsing. 
What structure signals are extracted 
From the repository tree, the system extracts: 
 Presence of layered architecture: 
o controllers/ 
o services/ 
o repositories/ 
 Separation of concerns 
 Test directory presence 
 CI/CD files (.github/workflows) 
 Containerization (Dockerfile) 
 Config hygiene (env.example, .gitignore) 
Example structure: 
src/ 
├── controllers/ 
├── services/ 
├── db/ 
│    └── migrations/ 
├── middleware/ 
├── tests/ 
Dockerfile 
This produces signals like: 
 layered_architecture = true 
 tests_present = true 
 container_ready = true 
These are binary or weighted signals, not opinions. 
How code is actually processed (important) 
Three-step approach (this is key) 
Step 1 — Deterministic scanning (no AI) 
 File names 
 Directory layout 
 File extensions 
 Commit metadata 
This is fast, reliable, and explainable. 
Step 2 — Targeted snippet extraction 
Instead of reading entire files: 
 Extract top N lines 
 Extract function signatures 
 Extract migration definitions 
 Extract config blocks 
This keeps cost low and avoids hallucination. 
Step 3 — Controlled LLM summarization 
LLMs are used only to: 
 Summarize what a snippet is doing 
 Classify pattern presence 
Example prompt: 
“Given this middleware snippet, does it implement rate limiting? Answer yes/no and why.” 
The LLM is not allowed to: 
 Rewrite code 
 Infer behavior beyond snippet 
 Make hiring decisions 
What the system explicitly does NOT try to do 
This is very important — and judges will respect this. 
It does NOT: 
 Run the code 
 Compile the code 
 Execute tests 
 Prove correctness 
 Detect bugs exhaustively 
 Replace code reviews 
Why? 
Because that would: 
 Be infeasible in a hackathon 
 Create false confidence 
 Violate explainability

“Ability to build scalable REST APIs in Node.js”
